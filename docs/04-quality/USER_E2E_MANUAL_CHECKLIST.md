- 아래 시나리오의 수치가 정확히 맞으려면 시작 전에 로컬 테스트 DB를 초기 상태로 맞춰야 한다.

## 2. 권장 준비 순서

1. 의존성 설치

```bash
pnpm install
```

2. 환경 변수 준비

```bash
cp apps/web/.env.example apps/web/.env
```

3. PostgreSQL 실행

```bash
docker compose -f infra/docker/docker-compose.local.yml up -d
```

4. 로컬 테스트 DB 초기화

주의:
- 아래 명령은 로컬 DB 데이터를 지운다.
- 수동 검증 전에 기대 결과를 고정하려면 실행을 권장한다.

```bash
pnpm -C apps/web exec prisma migrate reset --force
```

5. 개발 서버 실행

```bash
pnpm -C apps/web dev
```

6. 브라우저에서 열기

- [login](http://127.0.0.1:3000/login)

## 6. 상세 시나리오

### 시나리오 A. 비로그인 접근 차단 + 로그인

목적:
- 보호 라우팅이 동작하는지 확인
- 로그인 후 대시보드로 진입되는지 확인

사전 조건:
- 브라우저에 로그인 세션이 없어야 한다.

단계:
1. 브라우저에서 [dashboard](http://127.0.0.1:3000/dashboard)를 직접 연다.
2. `/login`으로 이동되는지 확인한다.
3. 이메일에 `guardian@example.com` 입력
4. 비밀번호에 `Guardian123!` 입력
5. `로그인` 버튼 클릭

기대 결과:
- 로그인 성공 후 `/dashboard`로 이동한다.
- 화면 상단에 `보호자 대시보드`가 보인다.
- 초기 상태 메시지 `아직 집계할 학습 데이터가 없습니다.`가 보인다.

### 시나리오 B. 문제집/시도/문항/오답 생성

목적:
- 핵심 입력 플로우가 실제 UI에서 저장되는지 확인

사전 조건:
- 시나리오 A 완료
- DB 초기화 직후라면 기대값이 가장 정확하다.

단계:
1. 상단 네비게이션에서 `기록 입력` 클릭
2. `문제집 제목`에 `실사용 수동 테스트 문제집` 입력
3. `출판사`에 `수동 테스트 출판사` 입력
4. `학년`은 `1`, `학기`는 `1학기` 유지
5. `문제집 저장` 클릭
6. 성공 메시지와 `materialId:`가 보이는지 확인
7. `시도 저장` 클릭
8. 성공 메시지와 `attemptId:`가 보이는지 확인
9. 첫 번째 문항 입력
   - 단원: `소인수분해`
   - 문항 번호: `1`
   - 난이도: `2`
   - 정답 여부: 체크하지 않음
   - 오답 메모: `첫 번째 오답`
10. `문항 저장` 클릭
11. 두 번째 문항 입력
   - 문항 번호: `2`
   - 난이도: `2`
   - 정답 여부: 체크하지 않음
   - 오답 메모: `두 번째 오답`
12. `문항 저장` 클릭
13. 세 번째 문항 입력
   - 문항 번호: `3`
   - 난이도: `3`
   - 정답 여부: 체크
   - 오답 메모: 비움
14. `문항 저장` 클릭

기대 결과:
- 첫 번째와 두 번째 저장 후 `문항과 오답이 저장되었습니다.`가 보인다.
- 오답 저장 후 `생성된 오답 ID:` 영역이 갱신된다.
- 세 번째 저장 후 `정답 문항이 저장되었습니다.`가 보인다.

### 시나리오 C. 중복 문항 번호 거부

목적:
- 같은 시도에서 중복 `problemNo` 저장이 막히는지 확인

사전 조건:
- 시나리오 B 완료

단계:
1. 현재 같은 입력 화면에서 문항 번호를 `3`으로 다시 입력
2. 단원은 `소인수분해` 유지
3. 정답 여부는 아무 값이나 가능
4. `문항 저장` 클릭

기대 결과:
- 오류 배너에 `problemNo already exists for attempt`가 표시된다.
- 추가 문항은 생성되지 않는다.

### 시나리오 D. 오답 목록 조회 + 카테고리 저장

목적:
- 오답 조회와 카테고리 저장이 동작하는지 확인

사전 조건:
- 시나리오 B 완료

단계:
1. `오답 관리 화면으로 이동` 클릭
2. 화면 상단에 `선택 학생: 기본 학생`이 보이는지 확인
3. `오답 목록 조회` 클릭
4. 첫 번째 오답 카드에서 `문제 잘못 읽음 (misread_question)` 체크
5. 첫 번째 오답 카드의 `카테고리 저장` 클릭

기대 결과:
- `오답 2건을 불러왔습니다.` 메시지가 보인다.
- `카테고리가 저장되었습니다.` 메시지가 보인다.
- 첫 번째 카드에 `현재 카테고리: misread_question`이 보인다.
- 두 번째 카드에는 여전히 `현재 카테고리: (없음)`이 보인다.

### 시나리오 I. 활성 학생 계정 reset + 새 초대코드 발급

목적:
- 이미 활성화된 학생 계정을 보호자가 끊고 새 초대코드로 다시 열 수 있는지 확인

사전 조건:
- 보호자 계정으로 로그인된 상태
- 학생 관리 화면에 `활성` 상태 학생이 1명 있어야 한다.
- 활성 학생이 없다면 먼저 `/students/manage`에서 초대코드를 발급하고, 다른 브라우저 또는 시크릿 창에서 `/student/activate`로 학생 계정을 1회 활성화한다.

단계:
1. 브라우저에서 [students/manage](http://127.0.0.1:3000/students/manage) 열기
2. `활성` 상태 학생 카드 확인
3. `계정 재설정 후 새 초대코드 발급` 클릭
4. 브라우저 confirm 대화상자에서 확인
5. 카드에 새 초대코드와 만료 시간이 표시되는지 확인
6. 같은 학생 카드 상태가 `미활성`으로 바뀌는지 확인
7. 시크릿 창 또는 다른 브라우저에서 기존 학생 `loginId`/비밀번호로 `/login` 로그인 시도

기대 결과:
- 보호자 화면에 `학생 계정을 재설정하고 새 초대코드를 발급했습니다.` 메시지가 보인다.
- 같은 학생 카드에 새 invite code가 표시된다.
- 카드 상태가 `활성`에서 `미활성`으로 바뀐다.
- 기존 학생 `loginId`로는 더 이상 로그인할 수 없다.

## 7. 빠른 실패 포인트

- 로그인이 안 되면:
  - seed 계정이 초기화되지 않았을 가능성이 높다.
  - `pnpm -C apps/web exec prisma migrate reset --force` 후 다시 시도

- 오답 목록이 2건이 아니면:
  - 시나리오 B 이전 데이터가 남아 있을 가능성이 높다.
  - DB 초기화 후 다시 시작

- 대시보드 수치가 문서와 다르면:
  - DB 초기화 없이 반복 실행했을 가능성이 높다.
  - 문항 수/오답 수/카테고리 수가 누적되었는지 먼저 확인

- 이미지 업로드가 전부 실패하면:
  - 실제 이미지 파일 대신 가짜 파일을 선택했는지 확인
  - `apps/web/public/uploads/wrong-answers` 또는 `apps/web/public/uploads/test-wrong-answers` 권한 문제 확인
